Architecture Blueprint v2.0

PriceWatch — Multi-Platform Price Comparator
E-Commerce + Quick-Commerce · India

Real-time price intelligence across Indian e-commerce and quick-commerce platforms. Crawls, normalises, and surfaces the best price with alerts, history, and an embeddable widget. Commodity coverage across three categories — platform-exclusive, QC-exclusive, and cross-listed (with time-to-deliver comparison).

Product Vision & Core Use Cases

🛒 Price Comparison

Search any product and see a live price table across all platforms, split by E-Com and Q-Com. One-click redirect to the cheapest source with a delivery-time indicator.

🔔 Price Alerts

Set a target price per product. System notifies via WhatsApp/email/push when price drops to threshold on any platform. Separate alerts for E-Com vs Q-Com if needed.

📈 Price + Time History

Price trend charts over 30/90 days. For cross-listed products (Category 3), a parallel track shows delivery time changes alongside price — is faster delivery getting cheaper?

🏷️ Category Browse

Browse by Dairy, Bakery, Snacks, Household, etc. Each category shows best-price platform and the category's commodity type distribution (E-Com only / QC only / Both).

📋 Basket Optimiser

Multi-item cart. System computes optimal split across 2–3 platforms minimising total spend + delivery fee. Factors in minimum-order thresholds per platform.

🔌 Embed Widget

JS snippet for blogs and review sites. Shows live price table per product. Configurable to show E-Com only, Q-Com only, or both. Ad/affiliate revenue model for publishers.

Two-stream architecture: The platform is built on a fundamental split between E-Commerce (Amazon.in, BigBasket — broad catalogue, standard delivery) and Quick-Commerce (Blinkit, Zepto, Swiggy Instamart, Country Delight — narrow catalogue, 10–30 min delivery). This split drives commodity categorisation, crawl strategy, and UI design throughout.
2

Top 10 Beeps — Best Price Signals Today

What is a "Beep"? A Beep is a best-price signal — a product where today's crawl found a price at or below a tracked threshold. Beeps are ranked by volume of user watchlists × price drop %. The top 10 Beeps scroll in the Global Focus Ticker at the top of every page.
Architecture note: Beeps are computed by the Alert Service after every crawl batch. The top-10 list is materialised into Redis with a 15-minute TTL. The ticker and this section both read from that Redis key — beeps:top10 — via the GET /v1/beeps/top10 endpoint. No DB hit per page load.

Commodity Categories — Three-Type Model

Two platform types, three commodity categories: Delivery time data is captured universally across all categories — the difference is how it is surfaced. In Category A & B, users choose their view via a dropdown (Rate / Time / Both). In Category C (cross-listed), both dimensions are always shown side-by-side by default since the trade-off between price and speed is the whole point.

API Architecture — Design Principles

RESTful + Event-driven hybrid: Public-facing endpoints are REST over HTTPS via AWS API Gateway. Internal service-to-service communication (crawler → alert service, price service → beep engine) uses Redis pub/sub and AWS SQS FIFO queues — not synchronous HTTP — to decouple write latency from user-facing reads.

⬛ Public Read APIs

All product search, price, and history endpoints are public (no auth). Rate-limited to 60 req/min per IP via API Gateway usage plan. Responses cached at CloudFront (15 min–1 hr) and Redis (5–30 min). These are the APIs consumed by the web app, mobile app, and embed widget.

REST / JSONCloudFront cached60 req/min/IP

⬛ Authenticated Write APIs

Alert creation/deletion, user profile, basket save — all require JWT (issued by Auth Service via magic-link or Google OAuth). Tokens are short-lived (15 min) with refresh tokens (7 days). No API key to manage for end users.

JWT RS256Magic link authGoogle OAuth2

⬛ Publisher / Widget API Key

Third-party publishers embedding the price widget get an API key (not JWT). Keys are rate-limited per domain (1,000 req/day free tier, paid tiers for higher volume). API key is scoped to read-only widget endpoints only.

API Key scopedJSONP + JSON

⬛ Internal / Admin APIs

Crawler status, normalisation queue, proxy health — internal admin APIs behind a separate admin JWT role. These endpoints are NOT exposed via CloudFront; they hit the load balancer directly and are IP-allowlisted to the VPN.

Crawler Engine — Two Tracks

E-Com vs Quick-Com require fundamentally different crawl strategies: E-Com sites (Amazon, BigBasket) are server-rendered or hydrated HTML — Scrapy + HTTP requests are sufficient, cheap, and fast. Quick-Commerce apps (Blinkit, Zepto, Instamart) are JavaScript SPAs with location-gating — require Playwright headless browser emulating a real device at a specific pin-code. Two separate worker fleets.

Crawl Lifecycle (both tracks)

1

Scheduler Triggers Job

EventBridge fires a cron per platform per crawl window. Emits a Celery task: { platform, track, category_url, pin_code, crawl_id }. QC tasks include pin_code; E-Com tasks do not.

2

Worker Initialises

Fargate container dequeues task. For QC track: selects residential proxy IP, randomises browser fingerprint, launches Playwright session with pin-code injection. For E-Com track: configures Scrapy spider with rotating User-Agent + cookie pool.

3

Scrape, Extract & Validate

Extracts: product_name, sku_id, price_inr, mrp_inr, discount_pct, in_stock, image_url, pack_size, delivery_eta_min. The delivery_eta_min field is populated for QC platforms only (e.g., Blinkit ETA shown in-app). Pydantic schema validation rejects malformed records.

4

Dedup, Normalise & Classify

Fuzzy match against product catalogue. Assign commodity_type (A/B/C) based on whether the canonical product already has listings on the other track. Newly seen products go to the normalisation queue for admin review.

5

Persist, Cache & Publish

Insert into price_snapshots. Update Redis latest-price key. Publish to price_update_channel for the Alert/Beep Service. Refresh the beeps:top10 key if any top-10 product was updated.

Adaptive Crawl Frequency — Lowest or Remove

Core rule (Req 5): A product's crawl frequency adapts to its price volatility. High-volatility products (frequent price changes) are crawled more often. Products that show no change over an extended window are either downgraded to the lowest tier, or — if absent from the platform for long enough — removed from the active crawl list entirely and flagged as "discontinued" in the catalogue.

Removal & User Notification

When a product is removed from a platform's active crawl, two things happen: (1) all active alerts for that product+platform pair are auto-triggered once with message "Product no longer available on [Platform] — price was ₹X when last seen." (2) The product detail page shows a "Last seen" badge instead of a live price. Historical data is kept indefinitely.

8

Anti-Block & Rate Management

🌐 Proxy Rotation

Residential proxy pool via BrightData. IPs rotate per request. Indian ISP targeting (Jio, Airtel) to avoid geo-blocks. QC crawlers use mobile-network proxies to mimic Android app traffic.

🎭 Browser Fingerprint

Playwright-Stealth injects: realistic navigator properties, WebGL renderer, canvas hash, WebRTC suppression, Chrome/Android user-agents. Fingerprints rotate per session.

⏱️ Human-Mimicry

Random delay 800–2400 ms between actions. Randomised scroll events. Mouse movement with Bézier curves. QC track also emulates app scroll-to-load for infinite product grids.

🍪 Session Pools

Per-platform cookie pools refreshed nightly. 5–10 dummy accounts per platform where login is required. Session rotation on detect signal.

🔁 CAPTCHA Handling

2Captcha / CapSolver for reCAPTCHA v2/v3 and hCaptcha. Auto-retry with fresh proxy. Exponential backoff on excessive CAPTCHA rate.

📊 Block Detection

Monitor for 403/429, bot-detection redirects, empty price grids. 3 failures → CloudWatch alert → ops Slack. Per-platform block rate tracked as a custom metric in Grafana.

⚠️ Legal Note: Crawling e-commerce platforms is legally grey. Review each platform's robots.txt and ToS before deploying. For Amazon, use the official PA-API as primary; Scrapy as fallback only. Rate-limit aggressively to avoid service disruption.

View: Key SQL patterns 
-- Upsert price snapshot
INSERT INTO price_snapshots
  (id, listing_id, price_inr, mrp_inr, discount_pct, in_stock, delivery_eta_min, crawled_at)
VALUES (gen_random_uuid(), $1, $2, $3,
   ROUND((($3-$2)::float/$3)*100)::int, $4, $5, NOW())
ON CONFLICT (listing_id, date_trunc('hour', crawled_at))
DO UPDATE SET price_inr=EXCLUDED.price_inr, in_stock=EXCLUDED.in_stock,
              delivery_eta_min=EXCLUDED.delivery_eta_min;

-- Adaptive tier update (run post-crawl per product per platform)
UPDATE platform_listings SET crawl_tier =
  CASE
    WHEN (SELECT COUNT(*) FROM price_snapshots ps2
          WHERE ps2.listing_id=pl.id
          AND ps2.crawled_at > NOW()-INTERVAL '5 crawls'
          -- simplified: use window function in production
          ) > 3 THEN 'hot'
    WHEN last_price_change < NOW()-INTERVAL '30 days' THEN 'minimum'
    WHEN last_price_change < NOW()-INTERVAL '7 days'  THEN 'slow'
    ELSE 'active'
  END
FROM platform_listings pl WHERE platform_listings.id = pl.id;

-- Category C: cross-listed view (both tracks present)
CREATE MATERIALIZED VIEW cross_listed_products AS
SELECT p.id, p.canonical_name, p.brand,
  MAX(CASE WHEN pl.platform_track='ecom' THEN ps.price_inr END) AS ecom_price,
  MAX(CASE WHEN pl.platform_track='qcom' THEN ps.price_inr END) AS qcom_price,
  MIN(CASE WHEN pl.platform_track='qcom' THEN ps.delivery_eta_min END) AS min_delivery_min
FROM products p
JOIN platform_listings pl ON pl.product_id=p.id
JOIN price_snapshots ps ON ps.listing_id=pl.id
  AND ps.crawled_at > NOW()-INTERVAL '8 hours'
GROUP BY p.id, p.canonical_name, p.brand
HAVING COUNT(DISTINCT pl.platform_track) = 2;

Secrets Management

All secrets (DB, proxy API keys, WhatsApp token, CAPTCHA solver key) in AWS Secrets Manager. Injected at ECS task launch via task definition secrets block. Never in env vars or repo.

Database Migrations

Flyway for versioned PostgreSQL migrations. Runs as one-off ECS task before each deploy. Backwards-compatible schema changes first, code deploy, then cleanup migration. Zero-downtime pattern.
